## はじめに

最近は AI エージェントにソースコードを書かせて開発するというスタイルも一般的になってきたかなと思います。皆さんはどうでしょうか。

私も AI エージェントに自然言語で指示を出し、ソースコードを書くというスタイルが定着してきて、自分でソースコードを書くという機会はほとんどなくなってきました。
ですが、最近いくつかの記事を読んで、エージェントに指示を出して開発をするスタイル、つまり「ソースコードを自分で書かない」というのはまだスタートラインにしか立っていないのではないかと思いました。

次のステップに進むためにはどのようなことをしないといけないのか、また世の中の先進的なテック企業はどのようなことをしているのか。気になったので調べてまとめてみました。

## 世界はどこまで進んでいるのか

調べてみると、世界のソフトウェア開発は「人間がコードを書く時代」から「人間が AI エージェントを指揮する時代」へ急速に移行しているようです。

- Google では新規コードの約50%が AI 生成
- Spotify ではシニアエンジニアが2025年12月以降1行もコードを書いていない
- OpenAI では1人のエンジニアが10〜20の並列エージェントを同時に操作している

これはもはや一部の先進企業の実験ではなく、産業構造の転換点と言えそうです。

OpenAI API プラットフォームのエンジニアリング責任者は、[Lenny's Podcast](https://www.lennysnewsletter.com/p/engineers-are-becoming-sorcerers) で「エンジニアは魔術師（Sorcerers）になりつつある」と言っています。
映画『魔法使いの弟子』のように、AI エージェントという「魔法のほうき」を指揮して仕事をさせるのがエンジニアの新しい姿だと。
同氏は「今後12〜24ヶ月は、エンジニアの役割が完全に変わる前に先行できる貴重な期間だ」とも言っています。

AI コーディングツール市場は2025年時点で73.7億ドル規模、2032年には301億ドルへ成長する見通しです。
GitHub Copilot は2,000万ユーザー超、Cursor は設立2年で ARR 10億ドル超を達成し、SaaS 史上最速の成長を記録しました。
開発者の84%が AI ツールを使用または導入予定であり、もう「使うかどうか」ではなく「どう使いこなすか」が問われるフェーズに入っています。

## 先進企業は具体的に何をしているのか

具体的にどんなことが行われているのか、いくつか例を挙げてみます。

### Spotify — Slack から通勤中にコード生成

Spotify は社内 AI システム「Honk」を構築しています。
これは Slack 経由で Anthropic の Claude Code に開発を指示する仕組みで、以下のようなワークフローを実現しています。

1. エンジニアがスマートフォンの通勤中にバグ修正や機能追加を指示
2. Claude Code が作業
3. 完了後 Slack で返答
4. 内容を確認して本番マージ

共同 CEO のソダーストロム氏は2026年2月の決算発表で「当社のシニアエンジニアは12月以降、自分ではコードを1行も書いていない。
彼らは今やコードを生成し、それを監督するだけだ」と言っています。
2025年には50以上の新機能をリリースし、同氏は「革命の始まりに過ぎない」とも言っています。

https://www.itmedia.co.jp/aiplus/articles/2602/13/news074.html

### OpenAI — 100% AI が書いた100万行のプロダクト

OpenAI のエンジニア Ryan Lopopolo 氏が公開した「Harness Engineering」は、特に印象に残った事例です。
3人のエンジニアが5ヶ月で約100万行のコードを含むプロダクトを構築しましたが、人間が直接書いたコードはゼロ行。すべて Codex エージェントが生成しました。
1人あたり1日平均3.5件の PR を処理し、従来の手動開発と比較して推定10倍の速度を実現しています。

このチームが提唱する「ハーネスエンジニアリング」のコア思想は、エンジニアの仕事はコードを書くことではなく、AI エージェントが効果的に仕事できる「環境（ハーネス）」を設計・構築することだという点にあります。具体的には以下の3つの柱で構成されています。

- **コンテキストエンジニアリング**: リポジトリ内のドキュメントを継続的に強化し、`AGENTS.md` は「目次」として約100行に抑え、詳細は `docs/` ディレクトリに構造化して配置する。「Slack の議論でチームがアーキテクチャに合意しても、エージェントが発見できなければそれは存在しないのと同じ」
- **アーキテクチャの制約**: 不変条件（invariants）を強制し、実装の詳細は指定しない。カスタムリンターや構造テストで監視する
- **ガベージコレクション**: エージェントが定期的にドキュメントの不整合やアーキテクチャ制約違反を検出・修正する

https://openai.com/index/harness-engineering/

### Anthropic — 67%生産性向上を数字で証明

Anthropic は132人のエンジニア調査、53件の定性インタビュー、20万件の Claude Code トランスクリプトを分析した内部調査を2025年12月に公開しました。
主要な数字は以下の通りです。

- エンジニアが日常業務の59%で Claude を使用（1年前の28%から倍増）
- 自己申告の生産性向上が50%（1年前は20%）
- Claude Code 導入後、1人あたりのマージ済み PR が67%増加
- Claude 支援業務の27%は「AI がなければ実行されなかったであろうタスク」（ダッシュボード作成、探索的作業など）
- Claude Code の自身のコードベースの90%は Claude Code 自身が書いた

一方で、エンジニアが「完全に委任」できるのは全体の0〜20%に過ぎず、80%以上は積極的な監督を必要とするとのことです。
AI は「常時協働者」であり、完全自律ではないという現実も報告されています。

https://anthropic.com/research/how-ai-is-transforming-work-at-anthropic

## 単一エージェントからマルチエージェントへ

調べていく中で、指示ベースの開発から次に進むうえで重要だと感じた概念が「マルチエージェント開発」です。
OpenAI 社内では1人のエンジニアが10〜20の並列 Codex エージェントを同時に操作し、コードレビューとマージに集中するワークフローが定着しています。
OpenAI のエンジニアリング責任者によれば、PR は70%増加し、レビュー時間は10〜15分から2〜3分に短縮されたとのことです。

### マルチエージェントの主要パターン

調べてみると、実践されているマルチエージェントパターンは主に5つあるようです。

| パターン | 動作 | 適用場面 |
| --- | --- | --- |
| Leader-Worker | オーケストレーターが専門ワーカーを生成 → 各自が作業 → リーダーが統合 | フルスタック機能開発（バックエンド、フロントエンド、テスト各担当） |
| Task Queue | リーダーがタスクキューを作成 → ワーカーが自己割り当て | 大規模マイグレーション、並列可能な単純作業 |
| Pipeline | Agent A → Agent B → Agent C と逐次処理 | データパイプライン、順序依存のある処理 |
| Competition | 複数エージェントが同一タスクに取り組み → 最良の解を採用 | アーキテクチャ設計の選択肢検討 |
| Watchdog | ワーカーが作業 → 監視エージェントがチェック → 問題時はロールバック | セキュリティクリティカルな変更 |

### 具体的なツールと実践方法

**Claude Code Agent Teams（実験的機能）** は、複数の Claude Code インスタンスがチームとして協調動作する仕組みです。
チームリードがタスクを割り当て、各チームメイトが独立したコンテキストウィンドウで作業し、ピアツーピアでメッセージングします。
ファイルロック機構（自己クレーム方式）で競合を防ぎ、`tmux` / `iTerm2` でスプリットペイン表示できます。

**Cursor 2.0** は Git worktree による分離環境で最大8エージェントを同時実行できます。
Background Agents を使えば、AWS の隔離マシン上でリポジトリをクローンし、別ブランチで自律的に機能を完成させてプッシュします。
開発者は10以上の並列エージェントを管理可能です。

**Claude-Flow（オープンソース）** は Claude Code 向けのオーケストレーションフレームワークで、ベクトルメモリ、ナレッジグラフ、スマートルーティングを備え、API コスト30-50%削減を実現します。

### マルチエージェントが有効な場面と不向きな場面

有効なのは並列可能な独立タスク（フロントエンド / バックエンド / テストの同時開発、複数仮説でのバグ調査、大規模リファクタ）のようです。
逆に、逐次依存するタスク、同一ファイルの編集、依存関係が複雑な作業ではオーケストレーションのオーバーヘッドが利益を上回るとのこと。
まずは2〜3並列から始めて慣れていくのが現実的なのかなと思います。

## エンジニアの役割はどう変わるのか

### Coder → Conductor → Orchestrator の進化

ESLint の作者が整理したエンジニアの役割進化がわかりすかったので紹介します。3つのフェーズで捉えられています。

| フェーズ | 時期 | 役割 | イメージ |
| --- | --- | --- | --- |
| 第1段階 Coder | 2024年前半 | AI は高機能なオートコンプリート。退屈なボイラープレートを書かせる | 便利なアシスタント |
| 第2段階 Conductor | 2024年後半〜2025年 | エンジニアが LLM に指示を出し、出力をレビューし、フィードバックする | オーケストラの指揮者 |
| 第3段階 Orchestrator | 2025年末〜現在 | 複数の自律バックグラウンドエージェントに作業を配分し、その出力を組み合わせる | NASA のミッションコントロール |

これを見ると「AI エージェントに指示を出してコードを書かせる」というスタイルは第2段階にあたります。
次に目指すのは第3段階、つまり複数のエージェントをオーケストレーションする段階ということになりそうです。

### 重要性が増すスキルと減るスキル

調べた中で、今後より重要になるとされているスキルは以下の通りです。

- システムアーキテクチャと設計（AI が実装できる形に問題を分解する力）
- AI 出力のレビュー・検証（微妙なロジックエラー、セキュリティ欠陥、アーキテクチャ反パターンを見抜く力）
- プロンプトエンジニアリングとコンテキスト管理
- テスト戦略の設計
- ドメイン知識とビジネス理解

Google のエンジニアは「最高のエンジニアは最速のコーダーではなく、AI を疑うべきタイミングを知っている人だ」と言っています。

一方、重要性が下がるスキルもあります。

- 構文や API 仕様の暗記
- ボイラープレート / CRUD コードの手書き
- 手動フォーマットやスタイル適用
- 定型的なデバッグ
- ルーチンドキュメントの執筆

これらはエージェントがほぼ完全に代替できる領域だと言われています。

### 「読まないコードを本番に出す」時代の品質管理

PSPDFKit 創業者の Peter Steinberger 氏は「I ship code I don't read（自分が読んでいないコードを本番に出す）」と言っています。
一見危険に聞こえますが、その背後には厳格なテスト・リンター・構造制約・レビュープロセスがあるようです。
AI が書いたコードの品質を担保するのは、もはや人間の目視レビューだけではなく自動化されたガードレール群だということですね。

OpenAI の「ハーネスエンジニアリング」が示すように、エージェントが苦戦したとき「もっと頑張れ」では解決にならず、「何の能力が欠けていて、それをエージェントにとって読み取れるかつ強制可能にするにはどうすべきか？」と環境を改善する発想が大事なのだと学びました。

## 落とし穴とリスク

とはいえ、AI エージェントでのコーディングは良いことばかりではなく、リスクについても調べてみました。

### セキュリティ

Veracode 社の2025年調査では、80のコーディングタスクと100以上の LLM を評価した結果、AI 生成コードのうちセキュアだったのはわずか55%でした。
しかもこの数字は新しい・大きなモデルでも改善していません。
よくある脆弱性は SQL インジェクション、ハードコードされたシークレット、安全でない Kubernetes 設定、存在しないパッケージの幻覚（サプライチェーン攻撃の温床）です。

### 技術的負債の加速

MIT Sloan Management Review は AI 生成コードを「技術的負債を蓄積する新しいクレジットカード」と言っています。
GitClear の調査では、AI ツール利用後にコードチャーン（追加後すぐに修正・削除されるコード）が倍増し、コードの再編成・再利用が減少する傾向が確認されています。
OX Security の報告では、AI 生成コードに共通する10のアンチパターン（リファクタの回避、過度の個別対応、既存ライブラリの再発明、意味のないテスト水増しなど）が特定されています。

### ラバースタンプ問題

Clutch 社の2025年6月調査では、開発者の59%が AI 生成コードを完全に理解しないまま使用しています。
Palo Alto Networks の研究では、AI 支援ありの開発者は AI なしの開発者より安全でないコードを書き、にもかかわらずそのコードが安全だと信じていた（自動化バイアス）とのことです。
これは個人の問題ではなく、組織的なガードレール設計の問題として捉えるべきだという指摘には納得しました。

### スキル萎縮

若手が AI に依存して基礎スキルを身につけられなくなるリスクは、Anthropic 内部でも懸念されています。
Google のエンジニアも「ジュニアのパイプラインを完全に切ると、5-10年後にリーダーシップの空白が生まれる」と警告しており、AI で得られた時間短縮を「もっと速くリリースする」ためだけに使わず、スケーラビリティ・セキュリティ・長期保守性に再投資する姿勢が大事だなと感じました。

## 次のステップに進むために

ここまで調べてきた先進企業の事例やリスクを踏まえて、指示ベースの開発から次に進むためにはどうすればよいか、ステップを整理してみました。

### Phase 1: 実験

まずは体験と基盤づくりから。AI エージェントの能力と限界を体感するところから始めるのがよさそうです。

#### `CLAUDE.md` / `AGENTS.md` / `.cursor/rules` を整備する

調べた中で、これが最も即効性がありそうだと感じました。リポジトリに、プロジェクトの構造、命名規則、アーキテクチャ上の不変条件、使用ライブラリとその理由を記述します。
OpenAI の知見に倣い、約100行の「目次」として設計し、詳細は `docs/` ディレクトリに構造化するのがよさそうです。
Slack やドキュメントにある暗黙知をリポジトリに集約することが、エージェント活用の出発点になりそうです。

#### 「spec → plan → implement → test → review」ワークフローを試す

Google Chrome チームのエンジニアが提唱するワークフローでは、まず AI と対話して仕様書（`spec.md`）を作り、次に段階的な実装計画（`plan.md`）を生成し、小さなチャンクずつ実装 → テスト → レビューするサイクルを回します。
「15分間のウォーターフォール」とも呼ばれ、巨大な指示を一度に投げるより格段に品質が高くなります。

#### 並列エージェントを2〜3台で始める

Claude Code の Sub-agents、Cursor の Background Agents、GitHub Copilot の Coding Agent など、非同期でバックグラウンド実行できるエージェントを低リスクなタスク（テスト追加、ドキュメント生成、依存関係更新）で試してみるのがよさそうです。

### Phase 2: 定着

ワークフローを標準化していくフェーズです。

#### AI コードレビューをパイプラインに組み込む

全 PR に対して AI エージェントによるコードレビューを自動実行するようにします。
人間のレビュー前に AI が一次レビューを行うことで、人間は高次の設計判断とビジネスロジックに集中できます。

#### テストを「安全ネット」として強化する

エージェントは優秀なテストスイートがある環境で劇的にパフォーマンスが上がります。
まずテストカバレッジが低い領域を AI に書かせ、人間がレビューして品質を担保します。
テストが充実すればするほど、エージェントに委任できる範囲が広がります。

### Phase 3: 最適化

マルチエージェント・オーケストレーションのフェーズです。

#### マルチエージェント開発を本格導入する

1つの機能開発で、バックエンド担当エージェント、フロントエンド担当エージェント、テスト担当エージェント、レビュー担当エージェントを並列実行する体制を構築します。
Git worktree や別ブランチでの分離が鍵です。

#### 「ハーネスエンジニアリング」の思想でリポジトリを設計する

すべての知識をリポジトリ内に格納し、カスタムリンターとアーキテクチャテストでエージェントの出力を制約し、ガベージコレクションエージェントで不整合を定期検出します。
エージェントが苦戦する箇所こそ、環境改善が必要なシグナルだと捉える発想が大切だなと思いました。

## おわりに

AI エージェントに指示を出してコードを書かせる。それ自体はもう当たり前になりつつあります。
でも今回いろいろ調べてみて、その先にはマルチエージェントのオーケストレーション、ハーネスエンジニアリング、自動化されたガードレールの設計といった世界が広がっていることを知りました。

OpenAI のエンジニアリング責任者が言うように、エンジニアの役割が完全に変わる前に先行できる貴重な期間は、まさに今この12〜24ヶ月です。
まずは `AGENTS.md` や `docs` の整備から始めてみる、1つのタスクでエージェントのバックグラウンド実行を試してみる。小さな一歩でも、踏み出すことが大事なのかなと思いました。
