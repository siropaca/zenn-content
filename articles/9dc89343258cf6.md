## はじめに

前回の記事では「ハーネスエンジニアリング」の思想を紹介し、AI エージェント駆動開発の次のステップとして「環境を設計する」ことの重要性について書きました。

https://zenn.dev/ryohei_and_idea/articles/9610bd5e378074

ハーネスエンジニアリングには3つの柱があります。

- **コンテキストエンジニアリング**: リポジトリ内のドキュメントを継続的に強化する
- **アーキテクチャの制約**: リンターや構造テストで不変条件を機械的に強制する
- **ガベージコレクション**: ドキュメントの不整合やアーキテクチャ制約違反をエージェントで定期検出・修正する

この中で最も即効性が高く、すぐに始められるのが「コンテキストエンジニアリング」です。  
今回はここにフォーカスし、AI エージェントのためのドキュメントをどう設計・構成すべきか、具体的な実践方法を紹介します。

`AGENTS.md` / `CLAUDE.md` には何を書くべきか、`docs/` ディレクトリにはどんな情報を置くべきか、AI がドキュメントを効果的に使うためのテクニック、そして具体的なディレクトリ構成例まで、読んだ方が自分のプロジェクトに導入できるレベルで書いていきます。

## なぜドキュメントを整備しないといけないのか

「ドキュメントは大事」と言われても、なかなか手が動かないのが正直なところだと思います。  
ですが、AI 駆動開発においてドキュメント整備は「あると嬉しい」ではなく「ないと動かない」レベルで重要です。

その理由を3つ挙げます。

### AI エージェントの精度はコンテキストの質で決まる

AI エージェントはコンテキストウィンドウに載った情報だけで判断します。  
どれだけ優秀なモデルでも、プロジェクト固有の設計方針や命名規則、ビジネスルールを知らなければ、推測で書くしかありません。

「よしなにやって」で期待通りの結果が出ないのは、エージェントの能力が低いのではなく、必要な情報がリポジトリに存在しないからです。  
ハーネスエンジニアリングの原則では「もっと頑張れ」ではなく「何の情報が不足しているか」を問えとされています。  
エージェントの出力品質が低いとき、改善すべきはプロンプトではなく環境（ドキュメント）です。

https://openai.com/index/harness-engineering/

### ドキュメントの整備度がエージェントの遵守率に直結する

ある分析では、200行以下の `CLAUDE.md` で92%以上のルール適用率を達成する一方、400行を超えると71%に低下するというデータがあります。

つまりドキュメントは「量」ではなく「構造と簡潔さ」が重要です。  
巨大な1ファイルにすべてを詰め込むのではなく、情報を適切に構造化し、必要なときに必要な分だけ読み込ませる設計が求められます。  
この考え方を「プログレッシブ・ディスクロージャー」と呼びますが、詳しくは後述します。

https://www.humanlayer.dev/blog/writing-a-good-claude-md

### 人間のためのドキュメントが AI のためのドキュメントにもなる

ドキュメントを整備するということは、チームの暗黙知を形式知にするということです。  
これは AI のためだけでなく、人間の新メンバーのオンボーディングにもそのまま効きます。

「ドキュメント整備のモチベーションが上がらない」という問題は多くのチームが抱えていると思いますが、AI 活用という強いインセンティブがその壁を越える原動力になります。  
AI のためにドキュメントを書いたら、結果的にチーム全体の生産性が上がった。そんな好循環が生まれるはずです。

## AGENTS.md と CLAUDE.md を理解する

AI エージェントへの指示ファイルとして代表的なものが `AGENTS.md` と `CLAUDE.md` です。  
まずはこの2つの違いと使い分けを整理します。

### AGENTS.md とは

`AGENTS.md` は Linux Foundation 傘下の Agentic AI Foundation が管理するオープン標準です。  
OpenAI Codex、GitHub Copilot、Cursor、Gemini CLI、Jules、Amp など20以上のツールが対応しており、特定のツールに依存しない「共通語」として位置づけられています。

リポジトリのルートに配置すると、対応するツールがこのファイルを自動的に読み込み、プロジェクト固有の指示としてエージェントに注入します。

https://agents.md/

### CLAUDE.md とは

`CLAUDE.md` は Anthropic の Claude Code 専用のメモリファイルです。  
セッション開始時にシステムプロンプトとして自動注入されます。

Claude Code では以下の6層のメモリ階層が用意されています。  
優先度の数字が小さいほど優先され、Managed CLAUDE.md（組織ポリシー）以外は、より具体的なファイルが上位を上書きします。

| 優先度 | ファイル | 用途 | バージョン管理 |
| --- | --- | --- | --- |
| 1 | Managed CLAUDE.md | 組織ポリシー（MDM 配布） | 対象外 |
| 2 | `~/.claude/CLAUDE.md` | ユーザー個人設定 | しない |
| 3 | プロジェクトルート `CLAUDE.md` | チーム共有の規約 | する |
| 4 | `CLAUDE.local.md` | 個人のプロジェクト固有設定 | しない |
| 5 | サブディレクトリ `CLAUDE.md` | モジュール固有の指示 | する |
| 6 | Auto Memory (`MEMORY.md`) | Claude 自身のセッションメモ | 自動管理 |

重要なのは**遅延読み込み**の仕組みです。  
起動時にはルートの `CLAUDE.md` のみがロードされ、サブディレクトリの `CLAUDE.md` はそのディレクトリのファイルを操作する時点で初めて読み込まれます。  
これにより、コンテキストウィンドウの浪費を防げます。

https://claude.com/blog/using-claude-md-files

### 両方を併用する戦略

2026年3月時点で Claude Code は `AGENTS.md` をネイティブサポートしていないため、両方を併用する戦略が最適です。

推奨アプローチは、`AGENTS.md` をプライマリとして記述し、`CLAUDE.md` から参照を設定する方法です。

```markdown:CLAUDE.md
AGENTS.md

<!-- 以下は Claude Code 固有の設定のみ記載 -->
```

こうすることで、Claude Code 以外のツール（Codex / Copilot / Cursor 等）を使うチームメンバーがいても、同じ指示が適用されます。

### ルートファイルに書くべき内容

ルートの `AGENTS.md`（または `CLAUDE.md`）は **200行以下**に抑えることが鉄則です。  
LLM のシステムプロンプトには既に多数の指示が含まれており、追加できる指示の「予算」は限られています。  
ルートファイルは「全部入りマニュアル」ではなく「ルーター（地図）」として設計するのがポイントです。

書くべき内容は以下の通りです。

```markdown
# プロジェクト名
TypeScript / Node.js プロジェクト。Backend(Express) + Frontend(React/Next.js)。

## コアコマンド
- 型チェック + Lint: `pnpm check`
- テスト実行: `pnpm test --run --no-color`
- 単一テスト: `pnpm test --run <path>.test.ts`
- ビルド: `pnpm build`

## プロジェクト構造
├─ src/api/       → Express API サーバー
├─ src/frontend/  → React フロントエンド
├─ docs/          → プロジェクトドキュメント

## 必須ルール（Do）
- 全関数に型アノテーションを付与する
- named export を使用する（default export は禁止）
- テスト追加はコード変更と同時に行う

## 禁止事項（Never）
- /migrations/ 配下のファイルを手動編集しない
- moment.js を使わない（date-fns を使用）
- any 型を使わない（unknown + 型ガードを使用）

## 確認が必要（Ask First）
- 新しい npm パッケージの追加
- データベーススキーマの変更

## ドキュメント参照
- アーキテクチャ全体: docs/architecture/overview.md を参照
- API 設計: docs/api/conventions.md を参照
- テスト戦略: docs/testing/strategy.md を参照
- DB 操作: docs/database/schema-overview.md を参照
```

なお、プロジェクト概要・コマンド・ディレクトリ構成などは `README.md` に既に書かれていることが多いです。  
AI エージェントは `README.md` も読めるため、重複する情報は `AGENTS.md` に書かず `README.md を参照` とだけ記載する方がコンテキストの節約になります。  
`AGENTS.md` にはエージェント固有のルール（Do / Never / Ask First）とドキュメントへのルーティングに絞るのが理想です。

https://devcenter.upsun.com/posts/why-your-readme-matters-more-than-ai-configuration-files/

`AGENTS.md` の運用で意識すべきポイントは以下の7つです。

1. **コード例のコピペではなくパスで参照**する（`docs/api/conventions.md を参照`）
2. **リンターで検査可能なルールは書かない**（リンター設定が正。二重管理を防ぐ）
3. **`/init` の自動生成結果をそのまま使わない**（生成された `CLAUDE.md` は手動でチューニングする）
4. **定期的にレビュー**する（`wc -l AGENTS.md` が200行超なら分割を検討する）
5. **コンパクション指示を含める**（「コンパクション時は変更ファイル一覧を維持」のように、コンテキスト圧縮時に残すべき情報を指定する）
6. **`CLAUDE.local.md` で個人設定を分離**する（gitignore 対象にし、チーム共有ファイルを汚さない）
7. **`.claude/rules/*.md` でパスベースのルールを分離**する（ルートファイルの肥大化を防ぐ）

記述スタイルについては「AI 向けドキュメント記述の原則」としてまとめて後述します。

## docs/ ディレクトリには何を置くべきか

コードからは「何が実装されているか」（What）は読み取れますが、「なぜそうしたか」（Why）は読み取れません。  
`docs/` ディレクトリは、AI エージェントの判断力を底上げするための知識ライブラリです。

具体的には、以下のようなドキュメントを配置します。

- **アーキテクチャ概要**: システム全体の鳥瞰図と制約
- **ADR**: 設計判断の記録と背景
- **ドメイン知識**: 用語集やビジネスルール
- **API 仕様 / DB スキーマ / コーディング規約**: 実装の前提となる技術的な取り決め
- **テスト戦略**: テストの粒度や境界の定義
- **実装パターン（patterns）**: リンターでは検査できない設計方針やコードパターン
- **機能仕様（specs）**: 機能単位の要件や設計

それぞれ見ていきます。

### アーキテクチャ概要

`docs/architecture/overview.md` にはシステム全体の鳥瞰図を配置します。  
ハーネスエンジニアリングでは「1000ページのマニュアルではなく地図を渡せ」と言われます。

ここで重要なのは、**「何が存在しないか」を明記する**ことです。

```markdown
## アーキテクチャの制約
- マイクロサービス間の直接通信は存在しない。すべてメッセージキュー経由
- フロントエンドから直接 DB にアクセスするパスは存在しない
- 認証は API Gateway で一元処理し、各サービスでは行わない
```

この「存在しないもの」の記述は、エージェントの実装を強く制約します。  
書かなければエージェントが「効率的だから直接通信しよう」と判断してしまう可能性があります。

### ADR（Architecture Decision Records）

ADR は「なぜその技術選定をしたか」「なぜその設計にしたか」を記録するドキュメントです。

AI エージェントは過去のチームの議論を知りません。  
ADR がなければ「なぜ moment.js ではなく date-fns なのか」「なぜ REST ではなく GraphQL なのか」を推測するしかなく、最悪の場合、過去に却下された選択肢を提案してきます。

```markdown:docs/adr/records/adr-0003-api-versioning.md
# ADR-0003: URL パスベースの API バージョニング採用

## Status: Accepted (2025-08-15)

## Context
API 互換性維持の戦略が必要。ヘッダーベース vs URL パスベースの2案を検討。

## Decision
URL パスベース（/api/v1/...）を採用。
理由: ルーティング層での分離が明確、CDN キャッシュとの相性が良い、
デバッグ時の視認性が高い。

## Consequences
- エンドポイント数が増えるが、バージョン間の差分が明確に管理可能
- 旧バージョンの廃止スケジュールを docs/api/deprecation-policy.md に記載
```

ADR は一度書いたら変更しません。  
意思決定が変わった場合は新しい ADR を作成し、古い ADR のステータスを `Superseded by ADR-XXXX` に更新します。

https://adr.github.io/

### ドメイン知識

AI エージェントが最も推測しにくいのがドメイン知識です。  
コードには直接書かれない暗黙のビジネスルールを明文化することが大切です。

特に効果的なのが**用語集**です。  
同じ概念に対してコード上の表現とビジネス上の呼称が異なる場合、エージェントは混乱します。

```markdown:docs/domain/glossary.md
# ドメイン用語集

| 用語 | 定義 | コード上の表現 |
| --- | --- | --- |
| メンバー | 有料プラン契約者 | `User` where `subscriptionStatus === 'active'` |
| ゲスト | 未登録・未ログインユーザー | `null` session |
| 割引適用期間 | 契約開始から12ヶ月間 | `discountEligibleUntil` field |
```

ビジネスルールやワークフロー（ユーザー登録フロー、決済処理フローなど）も、`docs/domain/` 配下に自然言語で記述しておくと、エージェントが実装時にこれらを参照して正確なロジックを書けるようになります。

### API 仕様 / DB スキーマ / コーディング規約

**API 仕様**は OpenAPI スキーマで管理するのが理想です。  
ポイントは**リアルなレスポンス例**を含めること。  
`"string"` や `123` のようなプレースホルダーではなく、実際のデータに近い例を書くと、AI のコード生成品質が大幅に上がります。

**DB スキーマ**はマイグレーションファイルだけでは不十分です。  
エンティティ間の関係性、制約条件の意図を自然言語で補足することが重要です。  
特に「`soft delete` されたレコードはクエリから除外する」のような暗黙のルールは、ドキュメントに書かなければエージェントは知りようがありません。

**コーディング規約**については、リンターで機械的に検査できるルールはドキュメントに書かないのがベストプラクティスです。  
リンター設定そのものを正とし、二重管理を防ぎます。  
ドキュメントに書くのはリンターでは検査できない設計方針に限定するのが良いです。  
具体的な実装パターンは量が多くなりがちなので、`docs/patterns/` に分離するのがおすすめです。

### 実装パターン（patterns）

`docs/patterns/` にはプロジェクト固有の実装パターンを配置します。  
リンターでは強制できないが、チームで統一したい書き方やアーキテクチャパターンが対象です。

```markdown:docs/patterns/error-handling.md
# エラーハンドリングパターン

## API レスポンスエラー
- ビジネスロジックエラーは `AppError` クラスを使い、HTTP ステータスとエラーコードを明示する
- 外部サービス（Stripe, SendGrid 等）のエラーは `ExternalServiceError` でラップする
- バリデーションエラーは Zod のエラーをそのまま返さず、`ValidationError` に変換する

## ロギング
- エラーログには必ずリクエスト ID を含める
- 外部サービスのレスポンスボディはログに残さない（個人情報リスク）
```

パターンが増えてきたら、ドメイン別（`api-patterns.md`、`db-patterns.md` など）にファイルを分割します。

### テスト戦略

`docs/testing/strategy.md` に Unit / Integration / E2E の境界定義やテストパターンを記述します。  
エージェントはテストスイートが充実した環境で劇的にパフォーマンスが向上します。  
テスト戦略が明文化されていれば、エージェントが適切な粒度のテストを書けるようになります。

### 機能仕様（specs）

`docs/specs/` には機能単位の要件や設計を配置します。  
Spec-Driven Development（SDD）と呼ばれるアプローチで、GitHub の [spec-kit](https://github.com/github/spec-kit) や AWS の [Kiro](https://kiro.dev/docs/specs/) が採用している考え方です。

`docs/` の他のディレクトリがシステム横断の共有知識を扱うのに対し、`specs/` は個別機能にフォーカスします。  
エージェントに機能の追加・修正を依頼する際、該当する spec を参照させることで実装精度が大きく向上します。

```markdown:docs/specs/payment.md
# 決済機能

## 要件
- クレジットカード・銀行振込・コンビニ払いに対応
- 決済完了後にメール通知を送信
- 未決済のまま24時間経過した注文は自動キャンセル

## 設計
- 決済処理は外部サービス（Stripe）に委譲
- Webhook で決済結果を受信し、orders テーブルのステータスを更新
- 自動キャンセルは cron ジョブで実行（毎時0分）
```

重要な設計判断が含まれる場合は、その背景を ADR として `docs/adr/` にも記録しておくと、知識が分散せずに済みます。

https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html

## AI エージェントがドキュメントを効果的に使うためのテクニック

ドキュメントの「内容」だけでなく、「構造」と「届け方」にも工夫が必要です。

### プログレッシブ・ディスクロージャー（3ホップナビゲーション）

すべてのドキュメントをコンテキストに載せることは不可能です。  
解決策は、**3ホップ以内で目的のドキュメントに到達するナビゲーション構造**を設計することです。

```
AGENTS.md（ルーター）
  → docs/INDEX.md（ドキュメントマップ）
    → docs/adr/INDEX.md（ADR インデックス）
      → docs/adr/records/adr-0003-api-versioning.md（個別ドキュメント）
```

すべての `docs/` サブディレクトリに `INDEX.md` を配置し、AI がそのディレクトリの「地図」を最初に読めるようにします。

```markdown:docs/INDEX.md
# ドキュメントマップ

タスクに関連するディレクトリの INDEX.md を最初に読み、
必要なファイルのみを参照すること。

| ディレクトリ | 内容 | いつ参照するか |
| --- | --- | --- |
| architecture/ | システム全体設計 | 新機能設計、構成理解時 |
| adr/ | アーキテクチャ意思決定記録 | 設計判断の背景を知りたい時 |
| api/ | API 仕様、規約、エラーコード | エンドポイント追加・修正時 |
| database/ | スキーマ設計、マイグレーション手順 | DB 操作時 |
| domain/ | ドメイン用語集、ビジネスルール | ビジネスロジック実装時 |
| domain-models/ | ドメインモデルの振る舞い・制約 | モデルの実装・修正時 |
| patterns/ | 実装パターン、設計方針 | コード実装時 |
| specs/ | 機能単位の要件・設計 | 機能の追加・修正時 |
| testing/ | テスト戦略、パターン | テスト追加・修正時 |
| standards/ | コーディング規約、Git 運用 | コード記述時全般 |
```

ここで重要なのは、`@docs/file.md` のようなインライン展開ではなく、**パスによる参照**（「`docs/file.md` を参照」）を使うことです。  
前者はファイル全文を毎回コンテキストに展開してしまいますが、後者はエージェントが必要なときだけ読みに行く遅延読み込みとなり、コンテキストウィンドウを節約できます。

### .claude/ ディレクトリの活用

Claude Code ユーザー向けですが、`.claude/` ディレクトリは `AGENTS.md` の肥大化を防ぐ強力な「逃がし先」です。

| ディレクトリ | 用途 | 読み込みタイミング |
| --- | --- | --- |
| `.claude/rules/*.md` | パスベースの条件付きルール | 該当パスのファイル操作時のみ |
| `.claude/skills/*/SKILL.md` | 参照知識（手順書、パターン集） | エージェントが必要と判断した時 |
| `.claude/commands/*.md` | カスタムスラッシュコマンド | ユーザーがコマンド実行時 |

特に `.claude/rules/` は強力です。  
YAML frontmatter でパスパターンを指定すると、そのパスのファイルを操作するときだけルールが読み込まれます。

```markdown:.claude/rules/database.md
---
paths:
  - "**/migrations/**"
  - "**/schema/**"
---

- マイグレーションファイルは必ず `bun run db:migrate:create` で生成する
- 手動でマイグレーションファイルを編集しない
- カラム削除は2段階で行う（1. コード側で未使用にする → 2. 次のリリースで削除）
```

`AGENTS.md` に全ルールを詰め込む代わりに、パスベースで分離することで200行以下を無理なく維持できます。

### AI 向けドキュメント記述の原則

`AGENTS.md` も `docs/` も含め、AI が読むドキュメント全般に共通する記述の原則です。  
AI はドキュメントをチャンク単位で処理するため、各セクションが独立して理解可能であることが重要になります。

1. **命令形・箇条書きで簡潔に書く**: 長文の段落は避け、スキャンしやすい構造にする
2. **最重要ルールを先頭に配置する**: LLM は冒頭と末尾の指示にバイアスがかかる
3. **禁止事項には代替案を必ず併記する**: 「X を使うな。Y を使え」のように書く
4. **各セクションを自己完結させる**: 他セクションへの暗黙の依存を排除する
5. **関連情報を近接配置する**: 同じ概念の情報を複数セクションに散在させない
6. **見出しに具体的なキーワードを含める**: 「設定」ではなく「PostgreSQL 接続設定」
7. **暗黙知を排除する**: 前提条件・前提知識を明示的に記述する
8. **一貫した用語を使う**: 「ユーザー名」と「ログインID」のような同義語の混在を避ける
9. **図表にはテキスト代替を付ける**: Mermaid 図にも自然言語の説明を併記する
10. **コード例を優先する**: 3段落の説明より1つの具体的なコードスニペットが効果的

### ドキュメント鮮度を維持する仕組み

ドキュメントは書いて終わりではなく、コードと一緒に更新し続ける必要があります。

CI でのドキュメント更新チェックがまず有効です。  
コード変更時に関連する `docs/` が未更新であれば Warning を出す GitHub Actions を設定できます。

また、`AGENTS.md` に**「アーキテクチャに影響する変更時は必ず ADR を作成する」**と記載しておくと、エージェント自身がドキュメント更新を促してくれます。

月次で `AGENTS.md` の行数をチェックし（`wc -l AGENTS.md`）、200行を超えていたら `.claude/rules/` や `docs/` への分割を検討するのもおすすめです。

## 具体的なディレクトリ構成例

ここまでの内容を踏まえて、具体的なディレクトリ構成例を2パターン紹介します。

### シングルリポジトリの場合

```shell
my-project/
├── AGENTS.md                    # メインルーター（200行以下）
├── CLAUDE.md                    # AGENTS.md 参照 + Claude 固有設定
│
├── .claude/
│   ├── settings.json            # プロジェクト権限設定
│   ├── rules/
│   │   ├── api.md               # paths: ["src/api/**"]
│   │   ├── database.md          # paths: ["**/migrations/**"]
│   │   └── testing.md           # paths: ["**/*.test.ts"]
│   ├── skills/
│   │   └── api-design/SKILL.md  # API 設計パターン参照知識
│   └── commands/
│       └── generate-adr.md      # /generate-adr コマンド
│
├── docs/
│   ├── INDEX.md                 # ドキュメントマップ
│   ├── architecture/
│   │   └── overview.md          # システム全体の鳥瞰図
│   ├── adr/
│   │   ├── INDEX.md             # ADR インデックス
│   │   ├── TEMPLATE.md          # ADR テンプレート
│   │   └── records/
│   │       ├── adr-0001-xxx.md
│   │       └── ...
│   ├── api/
│   │   ├── conventions.md       # エンドポイント命名・レスポンス形式
│   │   └── error-codes.md       # エラーコード一覧
│   ├── database/
│   │   ├── schema-overview.md   # ER 関係の自然言語説明
│   │   └── migration-guide.md   # マイグレーション手順
│   ├── domain/
│   │   ├── glossary.md          # ドメイン用語集
│   │   └── business-rules.md    # ビジネスルール
│   ├── domain-models/
│   │   ├── INDEX.md             # ドメインモデル一覧
│   │   ├── order.md             # 注文モデルの振る舞い・制約
│   │   ├── user.md              # ユーザーモデルの振る舞い・制約
│   │   └── ...
│   ├── patterns/
│   │   ├── error-handling.md    # エラーハンドリングパターン
│   │   └── ...
│   ├── testing/
│   │   └── strategy.md          # テスト戦略
│   ├── specs/
│   │   ├── payment.md           # 決済機能の要件・設計
│   │   ├── notification.md      # 通知機能の要件・設計
│   │   └── ...
│   └── standards/
│       ├── code-style.md        # コーディング規約
│       └── git-workflow.md      # ブランチ戦略・コミット規約
│
├── src/
├── package.json
└── ...
```

シングルリポジトリでは構造がフラットなので比較的シンプルです。  
ポイントは以下の通りです。

- `AGENTS.md` → `docs/INDEX.md` → 個別ドキュメントの3ホップナビゲーション
- `.claude/rules/` でファイルパスに応じた条件付きルールを分離し、`AGENTS.md` を軽く保つ
- 小規模プロジェクトでも `docs/` を構造化しておくことで、後からスケールしやすくなる

### モノレポの場合

```shell
my-monorepo/
├── AGENTS.md                    # メインルーター（200行以下）
├── CLAUDE.md                    # AGENTS.md 参照 + Claude 固有設定
│
├── .claude/
│   ├── settings.json
│   ├── rules/
│   │   ├── frontend-react.md    # paths: ["apps/frontend/**/*.tsx"]
│   │   ├── backend-api.md       # paths: ["apps/backend/src/**"]
│   │   ├── database.md          # paths: ["**/migrations/**"]
│   │   └── testing.md           # paths: ["**/*.test.ts"]
│   └── ...
│
├── docs/                        # 横断的・共有的な知識
│   ├── INDEX.md
│   ├── architecture/
│   │   └── overview.md          # 共通基盤 + システム全体の鳥瞰図
│   ├── adr/
│   │   ├── INDEX.md             # システム横断 ADR インデックス
│   │   ├── TEMPLATE.md          # ADR テンプレート
│   │   └── records/
│   │       ├── adr-0001-monorepo-adoption.md
│   │       └── ...
│   ├── api/
│   ├── database/
│   ├── domain/
│   ├── patterns/                # 実装パターン、設計方針
│   ├── specs/                   # 機能単位の要件・設計
│   ├── testing/
│   └── standards/
│
├── apps/
│   ├── frontend/
│   │   ├── AGENTS.md            # フロントエンド固有の指示
│   │   └── docs/
│   │       ├── INDEX.md
│   │       ├── tech-stack.md    # Next.js, React, Tailwind 等
│   │       ├── components.md    # コンポーネント設計パターン
│   │       └── adr/
│   │           ├── INDEX.md     # Frontend 固有 ADR（adr-f001〜）
│   │           ├── TEMPLATE.md
│   │           └── records/
│   │               └── ...
│   └── backend/
│       ├── AGENTS.md            # バックエンド固有の指示
│       └── docs/
│           ├── INDEX.md
│           ├── tech-stack.md    # Express, Prisma, Redis 等
│           ├── endpoints.md     # エンドポイント一覧
│           ├── domain-models/
│           │   ├── INDEX.md     # ドメインモデル一覧
│           │   ├── order.md     # 注文モデルの振る舞い・制約
│           │   ├── user.md      # ユーザーモデルの振る舞い・制約
│           │   └── ...
│           └── adr/
│               ├── INDEX.md     # Backend 固有 ADR（adr-b001〜）
│               ├── TEMPLATE.md
│               └── records/
│                   └── ...
│
├── packages/
│   └── shared/
│       ├── AGENTS.md
│       └── docs/
│
└── ...
```

モノレポでは「どの情報をどこに置くか」の判断が重要になります。  
原則は明確です。

- **横断的・共有的な知識はルート `docs/`**: アーキテクチャ、ADR、API 規約、DB 設計、ドメイン知識
- **コンポーネント固有の知識は `apps/*/docs/`**: フレームワーク固有のパターン、技術スタック

ルートの `AGENTS.md` はタスクベースのルーターとして設計し、具体的なドキュメントへ誘導します。

```markdown
## タスク別ドキュメントルーティング
- フロントエンドのコンポーネント作成 → apps/frontend/AGENTS.md を参照
- バックエンドの API エンドポイント追加 → apps/backend/AGENTS.md を参照
- 共有パッケージの修正 → packages/shared/AGENTS.md を参照
- データベーススキーマの変更 → docs/database/migration-guide.md を参照
```

モノレポでの反パターンとして、**各パッケージに詳細な `CLAUDE.md` を大量に配置するとメンテナンスが破綻する**ことが報告されています。  
ルートのルーターファイル + `.claude/rules/` によるパスベースのルール分離のほうが管理しやすく、現実的です。

## おわりに

AI 駆動開発において、ドキュメントはもはやコードの付属物ではなく、最も高レバレッジな成果物です。  
エージェントが見えない知識は存在しないのと同じです。  
逆に言えば、ドキュメントを整備するだけで、エージェントの出力品質は劇的に変わります。

すべてを一度に整備する必要はありません。  
まずはルートの `AGENTS.md` と `docs/architecture/overview.md` を書くところから始めてみるのがおすすめです。

また、今回いろいろと調査してみて感じたのは、この領域にはまだ確立されたベストプラクティスがないということです。  
この記事の内容もあくまで現時点でのひとつの型として捉えていただき、自分のプロジェクトに合わせて試行錯誤してみてもらえたらと思います。
